# Sub-Agent Transcript Projection Spec

Status: conceptual design draft.

Purpose: define how `codex-review-shell` should project Codex sub-agent activity
without flattening child-agent prompts and replies into the primary Codex chat.

message, and an `agentMessage` inside a sub-agent thread is not the primary
Codex answer.

## Problem

Current shell rendering is role-flat:

```text
ThreadItem.userMessage -> You
ThreadItem.agentMessage -> Codex
```

That is correct only for the primary Codex thread. It fails when a displayed
item belongs to a child agent thread:

```text
main agent prompt to child agent -> rendered as You
child agent answer -> rendered as Codex
```

The visible result is authority and authorship inflation:

- The operator appears to have sent prompts they did not send.
- The primary Codex agent appears to have produced answers generated by a child
  agent.
- The main implementation transcript stops being a truthful primary-thread
  conversation.

## Codex Protocol Evidence

Codex already carries the needed identity data.

`SessionSource` includes sub-agent sources:

```text
SessionSource::SubAgent(SubAgentSource::ThreadSpawn {
  parent_thread_id,
  depth,
  agent_path,
  agent_nickname,
  agent_role
})
```

App-server thread metadata includes:

```text
thread.source
thread.agent_nickname
thread.agent_role
```

Collab app-server items include:

```text
ThreadItem::CollabAgentToolCall {
  id,
  tool,
  status,
  sender_thread_id,
  receiver_thread_ids,
  prompt,
  model,
  reasoning_effort,
  agents_states
}
```

Supported collab tools:

```text
spawnAgent
sendInput
resumeAgent
wait
closeAgent
```

Vanilla TUI already treats this as multi-agent state. It renders compact events
such as spawned, sent input, waiting, resumed, and closed, and it hydrates
receiver metadata with `thread/read(include_turns = false)` when needed.

## Direct/OAI Boundary

The direct OAI path should not assume "sub-agent" is an upstream server
primitive. The upstream profile exposes turns, response items, tool calls, usage,
rate limits, and conversation continuity. Sub-agent identity is a harness or
controller object.

Implementation rule:

```text
Codex executable provider:
  derive sub-agent identity from app-server thread metadata and collab tool calls

Direct OAI provider:
  derive sub-agent identity from local harness AgentThreadRef / AgentRun records
```

Both providers should project into the same shell-level agent graph.

Thread identity is namespaced. A raw `threadId` is not enough because stored
sessions, different `CODEX_HOME` roots, direct-provider sessions, and future
multi-Codex panes may all produce ids that are only meaningful inside a runtime
scope.

Author identity is also first-class. Renderers must not decide authorship from
`role` alone. They render `TranscriptAuthorProjection.displayLabel` and style
from `TranscriptAuthorProjection.kind`.

## Projection Law

Primary Codex transcript:

```text
operator user message -> You
primary assistant final message -> Codex
primary assistant thought/tool/process items -> Thought process / process evidence
collab tool item -> Sub-agent activity
child thread userMessage -> not rendered as You in primary transcript
child thread agentMessage -> not rendered as Codex in primary transcript
```

Sub-agent transcript:

```text
senderThreadId prompt -> <sender agent label> -> <receiver agent label>
receiver answer -> <receiver agent label>
child agent thought/tool/process items -> scoped process evidence
operator direct intervention, if later supported -> Operator -> <agent label>
```

Only render `Main agent` or `Primary Codex` when
`senderThreadId === primaryThreadKey.threadId`. Nested agents must render
recursively:

```text
Primary Codex -> Robie [explorer]
Robie [explorer] -> Scout [worker]
```

Unknown identity rule:

```text
If receiver thread id exists but nickname/role are missing, render Agent <short id>
with unknown role. Do not fall back to You or Codex.

If thread context itself is unknown, render Unknown agent/source. Unknown identity
may be upgraded only by metadata or harness evidence, never by text heuristics.
```

Stored or child thread rows must never be replayed through live primary-thread
event handlers. They normalize into a presentation model, then render through
pure projection functions. This prevents child rows from mutating active turn
state, retry state, composer state, approval/request registries, runtime status,
or the opened primary thread id.

## Right Plane UX

The right plane should become a two-tab companion surface:

```text
ChatGPT | Sub-agents
```

`ChatGPT` remains the deliberation/review companion.

`Sub-agents` becomes a read-only v0 workbench for active and recent child-agent
threads attached to the currently open Codex primary thread.

Scope rule:

```text
The Sub-agents tab is scoped to the currently open Codex primary thread.
Switching primary Codex thread switches or clears the visible agent graph.
Late hydration results from a previous primary thread, graph revision, or
activation epoch are ignored.
```

Sub-agents tab layout:

```text
Sub-agents header:
  active count
  completed count
  errored count
  latest activity

Agent deck:
  one pane/card per child agent
  nickname / role / status
  parent thread id and child thread id in details
  compact transcript
  process evidence collapsed by default
```

Auto-switch policy:

```text
Auto-switch only when:
  first active sub-agent appears for the current primary thread
  ChatGPT composer is empty
  ChatGPT pane does not currently have focus
  user has not manually selected ChatGPT after the current graph revision
  no modal/menu/composer interaction is active

When additional sub-agents start, update badges and preserve the user's selected
right-plane tab unless attention is required. Do not auto-switch again for the
same graph revision.

When all sub-agents complete, keep the Sub-agents tab available but do not force
focus away from ChatGPT.
```

```ts
type RightPlaneInteractionState = {
  selectedTab: "chatgpt" | "sub_agents";
  userPinnedTab?: "chatgpt" | "sub_agents";
  chatgptComposerHasText: boolean;
  chatgptHasFocus: boolean;
  lastChatgptInteractionAt?: string;
  lastAutoSwitchGraphRevision?: number;
};
```

The middle plane should not own this in v0. Sub-agent conversations are companion
work streams, closer to the right-plane partner surface than to the middle-plane
project/control/workbench surface.

## Main Transcript Rendering

Main transcript collab summaries should follow the vanilla TUI semantic style:

```text
Spawned Robie [explorer]
Sent input to Robie [explorer]
Waiting for 2 agents
Finished waiting
Resumed Bob [worker]
Closed Bob [worker]
```

The main transcript may show:

- Collab action type.
- Target agent label.
- Status.
- Short prompt preview.
- Link/focus action to the Sub-agents tab.

Sub-agent activity should be distinct from normal tools and patches:

```text
Thought process / Process evidence
  Reasoning/commentary
  Shell / tool calls
  Patches
  Sub-agent activity
```

If a turn only has sub-agent activity and no reasoning/commentary, the parent
summary may render as `Sub-agent activity (N)` instead of implying hidden
reasoning.

The main transcript must not show:

- Full child-agent transcript as primary conversation.
- Child prompt as operator message.
- Child response as primary Codex final output.
- Full prompt bodies in compact summaries. Use preview text only; full prompt
  text appears in the right-plane details after explicit expansion.

## Transition Model

```text
CollabAgentToolCall received
  -> normalize collab event
  -> update AgentConversationGraph
  -> render compact main transcript activity
  -> enqueue bounded metadata hydration
  -> update right-plane Sub-agents tab badge
  -> auto-switch if policy permits
```

Child transcript loading:

```text
receiverThreadId discovered
  -> thread/read(receiverThreadId, include_turns = false) when supported
  -> hydrate nickname, role, source, status
  -> thread/read(receiverThreadId, include_turns = true) only when the
     Sub-agents tab/card is visible, expanded, or recently active
  -> build SubAgentTranscriptProjection from a presentation model
  -> render inside right-plane Sub-agents deck if graph revision still matches
```

Stored transcript loading:

```text
stored primary transcript loaded
  -> normalize stored rows into CodexMultiAgentTranscriptPresentationModel
  -> reconstruct AgentConversationGraph when evidence exists
  -> render main transcript summaries
  -> restore sub-agent deck if child transcripts can be found
```

Hydration rules:

```text
Hydration is concurrency-limited.
Hydration failures produce not_found/unavailable/stale states, not primary
transcript errors.
Hydration responses include graphRevision, activationEpoch, and primaryThreadKey.
Stale responses are ignored.
```

Collab normalization rules:

```text
Multi-receiver calls create one shared collab event and one edge/status update
per receiver. The main transcript renders one compact summary unless per-agent
status differs.

Nicknames are not identity. If labels collide, include role and short thread id:
Bob [worker] · a13f
Bob [reviewer] · 91c2

If a child/tool row cannot be confidently attached to a turn or agent, preserve
it as an orphan diagnostic item. Do not attach it to the nearest turn by text
heuristics.

Deduplicate only by stable ids/call ids/item ids or schema-equivalent adjacent
duplicates. Never deduplicate sub-agent messages by text alone.
```

Prompt association rule:

```text
If a child thread userMessage has no direct sender evidence, label it as
Parent agent -> Child agent using the child thread SessionSource parentThreadId.
If a matching collab call exists, attach it as supporting evidence.
If no parent metadata exists, label it Controller -> Agent <short id>, not You.
```

## Implementation Phases

Phase 1: Projection safety.

- Add `TranscriptAuthorProjection`.
- Stop rendering child-thread `userMessage` as `You`.
- Stop rendering child-thread `agentMessage` as primary `Codex`.
- Unknown child context renders as unknown, not primary.

Phase 2: Stored/live presentation model.

- Normalize live and stored rows into `CodexMultiAgentTranscriptPresentationModel`.
- Do not replay stored rows through live event lifecycle handlers.
- Preserve primary/sub-agent separation after restart.

Phase 3: Collab event normalizer and graph store.

- Add `AgentThreadGraphStore`.
- Normalize `CollabAgentToolCall` into graph edges and statuses.
- Preserve source/provenance and orphan diagnostics.

Phase 4: Metadata hydration.

- Hydrate receiver metadata with safe `thread/read(include_turns = false)` where available.
- Load child turns lazily with `thread/read(include_turns = true)` only when needed.
- Guard all hydration by primary thread key, graph revision, and activation epoch.
- Handle `not_found`, unavailable, and stale results without primary transcript errors.

Phase 5: Right-plane Sub-agents tab.

- Add `ChatGPT | Sub-agents` tabs.
- Render read-only child-agent panes.
- Add active/completed/errored badges.
- Add conservative auto-switch behavior.

Phase 6: Future controls.

- Optional focus/open child thread.
- Optional copy child transcript.
- Optional wait/close/resume controls only after app-server request semantics are
  safely mapped.
- Direct OAI harness alignment through local `AgentThreadRef` / `AgentRun`
  records.

## Acceptance Criteria

- Sub-agent prompts are never labeled `You`.
- Sub-agent outputs are never labeled as primary `Codex` final answers.
- The main Codex transcript shows primary conversation plus compact sub-agent
  activity summaries.
- The right plane exposes a Sub-agents tab with active child-agent panes.
- Unknown child metadata renders as unknown, not as user or primary agent.
- Stored/restarted transcripts preserve the same primary/sub-agent separation as
  live turns.
- ChatGPT tab state is preserved when Sub-agents appears.
- Auto-switch does not steal focus from active ChatGPT interaction.
- All sub-agent rendering is sourced from thread metadata, collab tool calls, or
  harness-owned agent records, not text heuristics.
- Direct OAI path models sub-agents as harness/controller objects rather than
  upstream server primitives.
- Thread identity is scoped by project, provider, source home, session file, and
  thread id where evidence exists.
- Child metadata hydration results are ignored when graph revision, activation
  epoch, or primary thread key no longer match.
- Multi-receiver collab calls update all target agent cards and produce one
  compact main transcript summary when statuses allow.
- Nested parent/child agent labels render recursively.
- Failed hydration does not break the primary transcript.
- Unknown agent identity can be upgraded by metadata evidence but never by text
  heuristics.
- Runtime reload preserves sub-agent separation for restored transcripts and
  does not reuse stale graph results from the old activation epoch.

## Fixture-Level Checks

- Primary thread spawns one child, sends prompt, waits, and child answers.
- Primary thread spawns two children with one multi-receiver wait.
- Nested child agent spawns a grandchild.
- Child metadata is missing nickname/role.
- Child thread is not found during hydration.
- Stored primary transcript has collab events but missing child session file.
- Direct OAI harness graph has `AgentRun` records but no upstream sub-agent
  primitive.
- Duplicate `event_msg` / `response_item` pairs are present.

Expected assertions:

- No child message renders as `You` or primary `Codex`.
- Unknown agent labels remain unknown until metadata evidence upgrades them.
- Multi-receiver events update all target agent cards.
- Nested parent labels are correct.
- Failed hydration does not break the primary transcript.

## Open Questions

- Does the shell app-server session currently receive child thread
  notifications automatically, or must it explicitly `thread/read` child thread
  ids discovered through collab items?
- Can every child thread be resolved from `threadId` plus current `sourceHome`,
  or do we need persisted `sessionFilePath` per `AgentThreadRef`?
- Should auto-switch happen for every new child agent or only the first active
  child agent per primary turn?
- Should the Sub-agents tab show completed child agents from previous turns by
  default, or only active/recent agents with an expandable archive?
- Should a future implementation allow direct operator messages to sub-agents,
  or should all sub-agent input remain mediated by the primary Codex agent and
  explicit controller transitions?
